1️⃣ ארכיטקטורה ותכנון 25 נקודות

האם הפרויקט מחולק לשכבות ברורות? האם כל שכבה יודעת רק את מה שצריך לה? מבנה טוב מאפשר לשנות חלק אחד מבלי לשבור את השאר.

📐 הפרדת שכבות (Layering)
50
האם הפרויקט מחולק ל-domain / application / infrastructure / CLI?
0: כל הקוד בקובץ אחד | 50: חלוקה חלקית ולא עקבית | 100: Clean Architecture — כל שכבה עצמאית
✓ Best practice: שכבת domain לא מייבאת כלום מ-infrastructure. תלויות זורמות פנימה בלבד.
עד 10 נק'
🔌 Dependency Injection (DI)
50
האם כל ה-wiring מתבצע במקום אחד (composition root)?
0: ישירות בכל מקום | 50: DI חלקי (חלק הוחלף, חלק ישיר) | 100: Container מרכזי — main.py לא יודע איך לבנות שירות
⚠ Anti-pattern: service = TransformationService(StrategyA()) בתוך main() — זו הדבקה ישירה. העבירו אותה לcontainer.

2️⃣ דיזיין פטרנים 30 נקודות

עיקר הפרויקט נסמך על פטרנים קלאסיים שנלמדו בשיעור: Strategy, Factory, Template Method, Decorator, Observer, Registry. הדגש הוא על שימוש נכון ומובן, לא על כמות.
עד 6 נק'
🎯 Strategy Pattern
50
Interface ברור + יישומים נפרדים לסוג-A וסוג-B?
0: אין / if-else ישיר | 50: strategy קיים אבל לא ניתן להרחיב | 100: הוספת סוג-C דורשת רק class חדש, ללא שינוי קוד קיים
✓ Open/Closed: אם הוספת variant חדש דורשת לפתוח קובץ קיים — זה לא strategy אמיתי.
עד 4 נק'
🏭 Factory Pattern
50
ParserFactory / StrategyFactory מחזירה instance/class נכון לפי סוג?
0: ללא factory | 50: factory בסיסי | 100: factory עם registry — ניתן להוסיף סוג ב-1 שורה
⚠ Anti-pattern נפוץ: factory שמחזירה instance במקום class. factory צריכה ליצור, לא לאחסן.
עד 4 נק'
📋 Template Method
50
BaseParser עם שלד פעולה + abstract methods שה-subclasses ממלאים?
0: כל parser עצמאי לחלוטין | 50: יש base class אבל אין @abstractmethod | 100: ABC + abstractmethod + ברור איפה כל subclass שונה
עד 7 נק'
🎁 Decorator Pattern
50
ValidatingStrategyDecorator עוטף strategy בלי לשנות אותו?
Decorator חייב לממש את אותו interface כמו ה-strategy.
0: אין Decorator | 50: Decorator קיים אבל לא עוטף נכון | 100: ניתן לעטוף עוד decorator בלי לגעת ב-strategy
✓ Best practice: ValidatingDecorator(strategy) — ה-decorator לא יודע אם ה-strategy הוא TypeA, TypeB, או decorator אחר.
עד 6 נק'
👀 Observer Pattern
50
האם ה-TransformationService מודיע ל-observers על אירועים?
LoggingObserver, ValidationObserver — ללא coupling.
0: אין observer — logging ישיר בכל מקום | 50: observer חלקי / צמוד מדי | 100: observers רשימה ב-constructor — הוספת observer ללא שינוי service
⚠ Anti-pattern: print(f"transformed {row}") בתוך strategy — זה coupling. ה-service לא אמור לדעת מי מקשיב.
עד 3 נק'
📦 Registry Pattern
50
RULES_REGISTRY / strategy_registry — מאגר מרכזי וגמיש?
0: אין / if-else לפי שם | 50: dict קטן | 100: registry מלא עם lookup ו-fallback

3️⃣ איכות קוד ובשלות הנדסית 25 נקודות

קוד טוב הוא קוד שניתן לקרוא, לבדוק, ולשנות בלי להיות מפחד. הסעיף הזה בודק: type safety, immutability, determinism, exception hygiene, וניהול הגדרות.
עד 5 נק'
🔤 Type Hints
50
האם כל signatures מסומנות? גם return types?
0: ללא type hints | 50: חלק מהפונקציות מסומנות | 100: 100% — כולל generics, Optional, Union
✓ Best practice: הריצו mypy --strict — אם עובר, זה 100.
עד 6 נק'
❄️ Immutable Dataclasses (frozen=True)
50
האם כל domain models ו-DTOs הם @dataclass(frozen=True)?
0: @dataclass רגיל — ניתן לשנות אחרי יצירה | 50: חלקם frozen | 100: כל DTOs frozen — mutation = TypeError בזמן ריצה
⚠ Anti-pattern נפוץ (83% מהפרויקטות!): @dataclass בלי frozen=True — נתונים שמשתנים בלא כוונה.
עד 5 נק'
🎲 Deterministic Randomness
50
הרצה זהה על אותו קלט → תמיד אותה תוצאה?
seed לפי תאריך / מזהה שורה.
0: אין randomness בכלל / תמיד שונה | 50: seed אחד גלובלי | 100: seed לפי שורה/תאריך — ניתן לבדוק בtest
✓ Best practice: random.seed(row.date.day + row.date.month*100) — כל שורה reproducible ועצמאית.
עד 4 נק'
⚡ Custom Exception Hierarchy
50
יש היררכיה ברורה? הודעות שגיאה מכילות context?
0: raise Exception("error") | 50: כמה exceptions אבל ללא היררכיה | 100: BaseError → ParseError, ValidationError, OutputError
עד 5 נק'
⚙️ Configuration Management
50
כל ה-thresholds, ranges, וכללים — בקובץ אחד?
0: hardcoded בכל קובץ | 50: config קיים אבל חלקי | 100: frozen RulesPerVariant ב-config/rules.py
⚠ Anti-pattern נפוץ: VALID_ENTRY_RANGE = (5, 11) ב-validating_decorator.py — שינוי כלל אחד דורש לחפש בכל הקוד.

🧪 Unit Tests
50
בדיקות ליחידות: parsers, strategies, decorators, validators.
0: ללא tests | 50: 1–3 קבצי test | 100: כל unit בודד + edge cases + coverage > 80%
✓ Best practice: deterministic randomness מאפשר לבדוק שווריאציה X של שורה Y תמיד תהיה Z. בלי seeding — לא ניתן לכתוב unit test על transformation!

50

⚠ ללא Docker הפרויקט לא ניתן להרצה בסביבה חיצונית — זו דרישה.

50

⚠ Anti-pattern: args = sys.argv[1:]; if "-o" in args: idx = args.index("-o") — ניהול ידני של CLI. השתמשו ב-argparse.

5️⃣ מודל אחיד לשני סוגי הדוח כלול בסעיפים לעיל

אחד האתגרים המרכזיים: שני סוגי דוח (Type A, Type B) שונים מאוד זה מזה. הפתרון הנכון — מודל פנימי אחיד שכולם עובדים עליו, ורק הparser והgenerator ספציפיים לסוג.
💡 שאלו את עצמכם: אם נוסיף סוג דוח שלישי, כמה קבצים חדשים צריך ליצור? כמה קבצים קיימים צריך לשנות?
תשובה אידיאלית: ליצור parser חדש + strategy חדש. לא לשנות כלום אחר.

6️⃣ ניהול זמן, תהליך ושיתוף פעולה — (כלול בבונוסים)

🔍 Code Review — חלק מרכזי בעולם העבודה האמיתי

ב-real-world software כל קוד עובר code review לפני merge. זהו כלי ללמידה, לגילוי באגים, ולהפצת ידע בצוות. בפרויקט הזה: ביצוע code review = בונוס + ניסיון.

נתתם הערות ספציפיות על קוד? מצאתם בעיות? הצעתם שיפורים?
קיבלתם הערות ספציפיות? שיפרתם קוד על פיהן?
השעות משמשות להבנת המאמץ — לא כ"ציון". אין תשובה "שגויה", היו כנים.
